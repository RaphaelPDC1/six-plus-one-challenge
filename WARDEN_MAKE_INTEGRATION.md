# Warden Make.com Integration Guide

This document describes how to run the 6+1 Warden through **Make.com only**, without Manus scheduled/background tasks. Make.com should act as the clock. The deployed app remains the judgment and delivery layer: it decides whether the Warden should speak, generates the message, sends it to the WhatsApp group through Whapi.Cloud, and logs the decision.

> **Operating rule:** Do not create new Manus scheduled tasks for routine Warden automation. They are not needed for this setup.

## Current Architecture

```text
Make.com schedule or event trigger
  ↓
Plain HTTP Warden endpoint on deployed app
  ↓
Challenge state + daily drama score
  ↓
Organic Warden window and daily cap checks
  ↓
Warden message generation when warranted
  ↓
Whapi.Cloud post to WhatsApp group
  ↓
Warden decision logged in the database
```

The app now exposes plain HTTP endpoints for Make.com so the scenario does not need to parse the nested tRPC response envelope. The older tRPC procedures still work and remain available as a fallback.

| Purpose | Recommended Plain HTTP Endpoint | tRPC Fallback |
|---|---|---|
| Scheduled Warden opportunity | `POST /api/warden/run` | `POST /api/trpc/warden.runCycle` |
| Immediate event trigger | `POST /api/warden/trigger` | `POST /api/trpc/warden.triggerImmediate` |
| Daily monitoring | `GET /api/warden/stats` | `GET /api/trpc/warden.getDailyStats` |
| Full state inspection | Use tRPC fallback | `GET /api/trpc/warden.getState` |

## Required Environment Variables

The deployed site must have the following environment variables configured. These are read by the server; they are not entered inside Make.com except for the Make webhook URL generated by Make.

| Variable | Required For | Notes |
|---|---|---|
| `WHAPI_TOKEN` | WhatsApp delivery | Whapi.Cloud bearer token used by the app to call `gate.whapi.cloud/messages/text`. |
| `WHAPI_GROUP_ID` | WhatsApp delivery | Target WhatsApp group ID, usually ending in `@g.us`. |
| `MAKE_WEBHOOK_URL` | App-to-Make event notification | Optional for scheduled morning messages, but required if the app should notify Make.com about event-based triggers. |

## Organic Warden Timing Model

The Warden should not post at a fixed time every day. The backend uses four organic windows, a deterministic randomized target minute for each day, one decision per window, late-window restraint, and a drama-driven daily cap.

| Window | UTC Time Range | Behaviour |
|---|---:|---|
| Morning | 07:00–09:00 | May speak if the morning data or carry-over story warrants it. |
| Midday | 12:00–14:00 | May speak if the group has produced useful mid-day evidence. |
| Evening | 18:00–20:00 | May speak as the day’s accountability picture becomes clearer. |
| Late | 21:00–22:00 | Only fires when there is enough drama or insight to justify a late intervention. |

The endpoint can return `success: true` while sending no WhatsApp message. Skip reasons such as `outside_organic_window_slot`, `late_window_without_enough_drama`, `Organic window already used today`, `Daily limit reached`, or `NO_MESSAGE decision` are healthy outcomes. They mean the Warden chose silence.

## Recommended Make.com Scenario: Scheduled Warden Windows

Create one Make.com scenario named **6+1 Warden Organic Windows**. Its only job is to probe the deployed app during the approved Warden windows. The app handles the actual send/no-send decision.

| Module | Make.com Module | Configuration |
|---:|---|---|
| 1 | Scheduler | Run every 15 minutes during the morning, midday, evening, and late windows. If one Scheduler cannot express all windows cleanly, create four scenarios or routes using the same HTTP module. |
| 2 | HTTP request | Method `POST`; URL `https://sixplusone-hpubf4au.manus.space/api/warden/run`; header `Content-Type: application/json`; body shown below. |
| 3 | JSON parse / condition | Treat `success: false` as an operational failure. Treat `success: true` with `messageSent: false` as a normal no-send decision. |
| 4 | Optional log/alert | Store the response in a Make Data Store, Google Sheet, or send an admin alert only when repeated failures occur. |

Use this request body in the HTTP module:

```json
{
  "source": "make-organic-window"
}
```

Use these probe times if your Make.com account requires separate schedules. The times below are **UTC**, matching the backend scheduler windows.

| Window | Make Probe Times |
|---|---|
| Morning | Every 15 minutes from 07:00 to 08:45 UTC |
| Midday | Every 15 minutes from 12:00 to 13:45 UTC |
| Evening | Every 15 minutes from 18:00 to 19:45 UTC |
| Late | Every 15 minutes from 21:00 to 21:45 UTC |

A successful message response is flat and easy for Make.com to parse:

```json
{
  "success": true,
  "messageGenerated": true,
  "messageSent": true,
  "message": "Day 12. Marcus lost a life before lunch. The scoreboard has started naming names.",
  "source": "make-organic-window",
  "timestamp": "2026-05-07T07:30:00.000Z"
}
```

A normal no-send response looks like this:

```json
{
  "success": true,
  "messageGenerated": false,
  "messageSent": false,
  "reason": "outside_organic_window_slot",
  "source": "make-organic-window",
  "timestamp": "2026-05-07T08:45:00.000Z"
}
```

## Optional Scenario: Immediate Event Trigger

Create a second Make.com scenario only if you want Make.com to cause immediate Warden comments for specific high-signal events. The endpoint still respects the daily Warden cap.

| Module | Make.com Module | Configuration |
|---:|---|---|
| 1 | Custom webhook or event source | Receive a life-loss, milestone, streak, or manual emergency event. |
| 2 | HTTP request | Method `POST`; URL `https://sixplusone-hpubf4au.manus.space/api/warden/trigger`; header `Content-Type: application/json`; body shown below. |
| 3 | JSON parse / condition | If `success: true`, no further WhatsApp module is needed. The app already posts to WhatsApp when `messageSent` is true. |

Example request body:

```json
{
  "triggerType": "life_loss",
  "context": {
    "participantName": "Marcus",
    "livesRemaining": 3,
    "timestamp": "2026-05-07T12:25:00.000Z"
  },
  "source": "make-event"
}
```

Valid trigger types are `life_loss`, `milestone`, `streak`, and `emergency`.

## Monitoring Endpoint

Use this endpoint from Make.com, a browser, or a manual HTTP client to check Warden usage for the day:

```text
GET https://sixplusone-hpubf4au.manus.space/api/warden/stats
```

Example response:

```json
{
  "success": true,
  "messagesSentToday": 1,
  "noMessageDecisions": 2,
  "dailyLimitHit": false,
  "dailyDramaScore": 6,
  "dailyMessageLimit": 4,
  "remainingMessages": 3,
  "timestamp": "2026-05-07T08:45:00.000Z"
}
```

## tRPC Fallback Endpoints

The original tRPC endpoints are still suitable for Make.com if you prefer not to use the plain wrapper. The local endpoint probe confirmed that both a plain body and a tRPC-style `json` envelope are accepted for `warden.runCycle`, but the response is nested under `result.data.json`, which is why the plain HTTP wrapper is recommended.

| tRPC Endpoint | Method | Body |
|---|---|---|
| `https://sixplusone-hpubf4au.manus.space/api/trpc/warden.runCycle` | `POST` | `{"source":"make-organic-window"}` or `{"json":{"source":"make-organic-window"}}` |
| `https://sixplusone-hpubf4au.manus.space/api/trpc/warden.triggerImmediate` | `POST` | `{"triggerType":"life_loss","context":{},"source":"make-event"}` |
| `https://sixplusone-hpubf4au.manus.space/api/trpc/warden.getDailyStats` | `GET` | No body |
| `https://sixplusone-hpubf4au.manus.space/api/trpc/warden.getState` | `GET` | No body |

## Manual Tests

After the next deployment, test the recommended endpoint with:

```bash
curl -X POST https://sixplusone-hpubf4au.manus.space/api/warden/run \
  -H "Content-Type: application/json" \
  -d '{"source":"manual-test"}'
```

Test the stats endpoint with:

```bash
curl https://sixplusone-hpubf4au.manus.space/api/warden/stats
```

Test an immediate trigger only if you are comfortable potentially sending a Warden message to the live WhatsApp group:

```bash
curl -X POST https://sixplusone-hpubf4au.manus.space/api/warden/trigger \
  -H "Content-Type: application/json" \
  -d '{"triggerType":"emergency","context":{"note":"Manual Make.com test"},"source":"manual-test"}'
```

## Error Handling in Make.com

Make.com should treat `success: false` as an operational failure and alert the admin if it repeats. Make.com should treat `success: true` with `messageSent: false` as normal. That is usually the Warden conserving its voice.

| Response Condition | Make.com Action |
|---|---|
| `success: true`, `messageSent: true` | Log success. Do not send another WhatsApp message from Make.com. |
| `success: true`, `messageSent: false` | Log optionally. No alert needed. |
| `success: false` | Retry once after a short delay and alert the admin if repeated. |
| HTTP/network failure | Retry according to Make.com’s normal error handler and alert the admin if repeated. |
